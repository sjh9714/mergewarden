## Fact Sheet

- Category: checkout-free change-control layer for AI-generated PRs.
- Runtime: deterministic TypeScript; no LLM calls.
- Trust boundary: exact base-commit policy; no checkout or PR code execution.
- First use: public PR scan through `npx`.
- Evidence: rule ID, severity, path, canonical evidence, finding ID, policy
  digest, base/head SHA, and collection completeness.
- Limitation: not a semantic code reviewer or complete workflow taint analyzer.

## Core Story

An AI agent can correctly edit the requested documentation and still change a
release workflow, agent instructions, MCP configuration, or package lifecycle
script in the same PR. Tests and semantic review answer whether code works;
Agent Gate records whether the change crossed an agreed repository boundary.

## Show HN Draft

**Title:** Show HN: Agent Gate – a checkout-free policy gate for AI-generated PRs

AI coding agents can open useful PRs, but ordinary tests do not tell you when a
PR also expands workflow permissions, changes agent instructions, or edits
files outside its declared task. Agent Gate reads PR metadata through GitHub
APIs, loads policy from the exact base commit, and produces deterministic
evidence without checking out or executing PR code.

You can try a public PR with:

```bash
npx --yes @jinhyuk9714/agent-gate@0.3.1 scan owner/repo#123
```

I would value feedback on the policy model, false positives, and evidence that
would make maintainers comfortable moving from warn to block.

## Ten Researched Design-Partner Drafts

These drafts use public workflow facts checked on 2026-07-10. Re-open the linked
workflow and personalize the maintainer name immediately before sending. Send
individually only after approval; do not bulk-send, request stars, or imply an
endorsement.

### 1. openclaw/openclaw

Your [Docs Agent workflow](https://github.com/openclaw/openclaw/blob/266ca5b3a2e094ac73f2820563385e57add0e6df/.github/workflows/docs-agent.yml)
already has unusually careful controls: exact-main gating, full-SHA Action
pins, a docs-only patch check, and a bounded cadence. I am building Agent Gate
to turn similar boundaries into deterministic PR evidence without checkout.
Would a report against one public agent-generated PR be useful for comparing
its signal with your existing safeguards?

### 2. matomo-org/matomo

Matomo's [Codex Review workflow](https://github.com/matomo-org/matomo/blob/21459956ff1a0db1b8bf62d5cf71199537ca91dc/.github/workflows/codex-review.yml)
already stops when reviewer-automation files change and separates trusted base
assets from the PR checkout. Agent Gate addresses the adjacent question: did an
AI PR cross declared scope or expand workflow/control-plane privileges? I would
value a false-positive review of a public report; no installation is required.

### 3. ComposioHQ/composio

The [Changelog → Docs PR workflow](https://github.com/ComposioHQ/composio/blob/2fcdc43bf9977171095726dc6eba40bd5433a728/.github/workflows/docs.changelog-to-docs.yml)
uses a SHA-pinned Codex Action with `contents` and `pull-requests` write access
to produce documentation PRs. Agent Gate is designed to surface future changes
to exactly those permission, pinning, and intended-path boundaries. Would a
read-only report against a public PR help evaluate whether that evidence is
useful or noisy?

### 4. 567-labs/instructor

Instructor's [OSS Issue Tagger](https://github.com/567-labs/instructor/blob/cd698be4e79b1b3aa64da31d9e551c4e8f60be11/.github/workflows/oss-issue-tagger.yml)
separates the Codex suggestion job from an issue-writing job and currently
guards the latter behind a shadow condition. Agent Gate could make changes to
that phase boundary, token permission, or Action pin visible as PR evidence.
Could I share one public scan for policy and false-positive feedback?

### 5. ton-blockchain/ton

TON's [Codex auto-review workflow](https://github.com/ton-blockchain/ton/blob/6308c96bc8c4e95d4d8c598a84d648534331e92b/.github/workflows/codex-review.yml)
passes PR title/body context into a SHA-pinned review Action and posts with
write permissions in a separate job. Agent Gate v0.3 includes a deliberately
narrow untrusted-prompt rule plus workflow permission-delta evidence. I would
appreciate feedback on a public report, especially where the rule should stay
silent.

### 6. agentclientprotocol/codex-acp

The [Codex Update workflow](https://github.com/agentclientprotocol/codex-acp/blob/8aff492d4b033ff2c02ad3b9d591994d57617463/.github/workflows/codex-update.yml)
uses a GitHub App token, lets Codex finalize an update, pushes a branch, and
opens a PR. Agent Gate is intended to record when future PRs alter those
agent-control-plane and workflow-capability boundaries. Would a no-install
public scan be useful for checking whether the evidence matches your review
model?

### 7. igorescodro/alkaa

Alkaa's [Claude workflow](https://github.com/igorescodro/alkaa/blob/8524e062027ca8f9766cc8bf434a13ffdcbfdb97/.github/workflows/claude.yml)
limits comment/issue triggers to the maintainer account and keeps repository
permissions mostly read-only, while granting OIDC for the agent Action. Agent
Gate can report permission or Action-reference changes without executing the
workflow. Could I send a public report to learn whether those findings are
appropriately scoped?

### 8. slhck/ffmpeg-normalize

The [Claude Code workflow](https://github.com/slhck/ffmpeg-normalize/blob/a329700dd410e4a0c4f9af6725167f7a76824d90/.github/workflows/claude.yml)

curated tool allow-list. Agent Gate is built to make later permission, prompt,
and pinning changes explicit at PR review time. I would value a maintainer's
assessment of one public scan's useful signal versus noise.

### 9. MervinPraison/PraisonAI

PraisonAI's [Gemini Issue Review workflow](https://github.com/MervinPraison/PraisonAI/blob/b146910fafbe8ded6aa1cbffe30aabc9e8f48abb/.github/workflows/gemini-issue-review.yml)

branch, commit, push, and open a PR. That is close to the concrete boundary our
narrow agentic-workflow rule is meant to explain—not a claim that the workflow
is vulnerable. Would a read-only public report help test the evidence and
remediation wording?

### 10. clouddrove/github-shared-workflows

Your [reusable Gemini PR review workflow](https://github.com/clouddrove/github-shared-workflows/blob/f6ef7e54f3a1f4e2a05a66ed1a8702c07ae94346/.github/workflows/pr-gemini-review.yml)
accepts a caller-provided review prompt, computes a PR diff, and posts the model
summary with pull-request write permission. Agent Gate can surface changes to
the prompt sink, reusable-workflow pinning, and effective permissions as
deterministic evidence. Could I share one public scan for feedback before
recommending any installation?
